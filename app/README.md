## Admin RBAC (登录/登出)

Back office now requires session login and RBAC permissions.

Default demo accounts:

1. `admin / admin123` (`super_admin`)
2. `ops / ops123` (`ops_admin`)
3. `auditor / audit123` (`auditor`)
4. `skill / skill123` (`skill_admin`)

Production recommendation:

1. Do not keep plaintext demo passwords.
2. Provide users via `AUTH_USERS_FILE` or `AUTH_USERS_JSON` with `passwordHash` (generated by `npm run auth:hash`, default algorithm: `scrypt-v1`).
3. Set `AUTH_PASSWORD_PEPPER` in environment.
4. In production, enable strict mode: `AUTH_REQUIRE_EXTERNAL_USERS=1` (or run with `NODE_ENV=production`, which enables strict mode by default).
5. Enable login safety controls:
   - `AUTH_MAX_FAILED_ATTEMPTS` (default 5)
   - `AUTH_LOCKOUT_MS` (default 900000)
   - `AUTH_FAIL_WINDOW_MS` (default 600000)
6. Session strategy controls:
   - `AUTH_SESSION_SLIDING=1|0` (default 1)
   - `AUTH_SINGLE_SESSION_PER_USER=1|0` (default 0)

Production auth hardening:

1. `AUTH_REQUIRE_EXTERNAL_USERS=1`：必须使用外部账号源（文件或 JSON）
2. `AUTH_FORBID_DEMO_USERS=1`：禁止 `admin/ops/auditor/skill` 演示账号
3. `AUTH_PASSWORD_PEPPER`：生产环境必填（服务启动会校验）
4. `CORS_ALLOW_ORIGIN`：生产环境必填，且不能是 `*`

## Core Rules

1. One human creator can create only one parent digital employee.
2. Parent digital employee auto-manages child agents during task execution.
3. Skills are auto-sedimented by digital employees during work.
4. OSS candidate search is auto-triggered by digital employees on correction/failure flow.

## Front Tenant Isolation (账号/租户隔离)

用户端 API 采用账号/租户上下文隔离：

1. 所有 `/api/front/*` 与 `/api/tasks/*` 请求必须携带：
   - `x-tenant-id`
   - `x-account-id`
2. 服务端按 `(tenantId, accountId)` 进行资源隔离，跨租户访问返回 not found / forbidden。
3. `Employee` 与 `Task` 记录会持久化 `tenantId/accountId`，并写入审计事件。

## External Write Contract (Phase 2)

When creating a task that writes to real enterprise systems, include `externalWrite` in `POST /api/front/tasks` payload:

```json
{
  "employeeId": "employee-id",
  "goal": "Update ERP payment status",
  "externalWrite": {
    "system": "erp",
    "operation": "payment.update",
    "resource": "payment",
    "idempotencyKey": "erp-payment-20260216-001",
    "compensation": {
      "action": "payment.revert"
    }
  }
}
```

Rules:

1. Duplicate idempotency key under same `employee + system + operation` is rejected.
2. If rollback is triggered and compensation is configured, system emits `integration.compensation.queued`.
3. Compensation queue is processed automatically. On success/failure it emits:
   - `integration.compensation.running`
   - `integration.compensation.succeeded`
   - `integration.compensation.retry_scheduled`
   - `integration.compensation.dead_lettered` (after max retries)
   - `integration.compensation.deferred`

Retry and dead-letter controls:

```bash
COMPENSATION_MAX_ATTEMPTS=3
COMPENSATION_BACKOFF_MS=1000
```

Manual retry API (admin auth required):

```bash
POST /api/admin/tasks/:taskId/retry-compensation
```

Enterprise adapter environment:

```bash
ENTERPRISE_SYSTEM_BASE_URL=http://127.0.0.1:19090
ENTERPRISE_COMPENSATION_PATH=/api/compensations
ENTERPRISE_SYSTEM_API_KEY=
ENTERPRISE_SYSTEM_TIMEOUT_MS=8000
```

Self-bootstrap loop is automatic in server runtime:

1. Task executor ticker (`1s`)
2. OSS research ticker (`2.5s`)
3. Bootstrap governance ticker (`5s`) with auto-advance/corrective/manual-review trigger
