# 数字员工研发拆解清单（自举自动执行）

## 1. 执行原则

1. 采用自举循环自动推进：`任务执行 -> 评估 -> 纠偏 -> 沉淀 -> 再执行`
2. 以结果交付为导向，按阶段直接产出文档与实现清单
3. 默认不做过程确认，出现阻塞再单点上报

## 2. 目标范围（当前批次）

1. 建立 MVP 到规模化的技术拆解基线
2. 明确 OpenHands + AgentScope 的工程落地边界
3. 输出可直接进入研发排期的服务/API/数据/埋点清单

## 3. 服务拆解（P0/P1/P2）

### 3.1 P0 服务（MVP 必做）

1. `identity-service`
   - 数字员工身份、岗位归属、账号生命周期
2. `policy-service`
   - 权限策略、红线规则、审批策略
3. `orchestrator-service`（AgentScope 适配）
   - 父Agent编排、子Agent调度、任务状态机
4. `execution-service`（OpenHands 适配）
   - 执行动作、命令白名单、沙箱运行
5. `audit-service`
   - 决策链、调用链、结果链统一落库
6. `evaluation-service`
   - 目标达成率、错误复发率、Skill复用率计算

### 3.2 P1 服务（MVP 后）

1. `skill-registry-service`
   - Skill 注册、版本、灰度、退役
2. `error-learning-service`
   - 错误分级、隔离池、负样本转化、纠偏任务派发
3. `oss-governance-service`
   - 开源准入、许可证校验、漏洞与维护状态

### 3.3 P2 服务（规模化）

1. `capability-market-service`
2. `cross-department-routing-service`
3. `roi-analytics-service`

## 4. 核心 API 清单（首版）

### 4.1 身份与岗位

1. `POST /employees` 创建数字员工
2. `POST /employees/{id}/onboard` 上岗
3. `POST /employees/{id}/offboard` 停岗
4. `PUT /employees/{id}/role-contract` 更新岗位合同

### 4.2 编排与执行

1. `POST /tasks` 创建任务
2. `POST /tasks/{id}/dispatch` 父Agent分派子任务
3. `POST /tasks/{id}/execute` 执行层触发动作
4. `POST /tasks/{id}/rollback` 回滚
5. `POST /tasks/{id}/abort` 熔断/停机

### 4.3 治理与审计

1. `POST /approvals` 发起高风险审批
2. `POST /approvals/{id}/decision` 审批决策
3. `GET /audits/{task_id}` 查询全链路审计
4. `POST /risk/events` 上报风险事件

### 4.4 评估与成长

1. `POST /evaluations/run` 启动周期评估
2. `GET /evaluations/{period}` 获取评估结果
3. `POST /skills/promote` Skill 转正
4. `POST /skills/retire` Skill 退役

## 5. 数据模型（核心表）

1. `digital_employees`
   - `id, employee_code, dept_id, role_id, status, owner_ids, created_at`
2. `role_contracts`
   - `employee_id, responsibilities, forbidden_rules, kpi_json, risk_level`
3. `tasks`
   - `id, employee_id, parent_task_id, state, priority, trace_id, created_at`
4. `task_actions`
   - `task_id, action_type, input_ref, output_ref, status, duration_ms`
5. `agent_runs`
   - `task_id, parent_agent_id, child_agent_id, engine, run_state`
6. `approvals`
   - `id, task_id, policy_id, approver_id, decision, decided_at`
7. `audit_logs`
   - `trace_id, task_id, actor_id, event_type, payload_ref, timestamp`
8. `error_cases`
   - `id, task_id, severity, root_cause, isolated, replayable`
9. `skills`
   - `id, name, version, status, reuse_count, rollback_count`
10. `evaluation_reports`
   - `period, employee_id, outcome_score, error_recurrence, skill_reuse, total_score`

## 6. 埋点与可观测（统一字段）

1. 强制字段：`trace_id, task_id, employee_id, parent_agent_id, child_agent_id`
2. 关键指标：
   - `task_success_rate`
   - `p1_incident_count`
   - `error_recurrence_rate`
   - `skill_reuse_rate`
   - `cost_per_task`
3. 告警规则（首版）：
   - P1 > 0 立即告警
   - 连续失败率超阈值触发熔断
   - 越权尝试触发降级

## 7. 双引擎适配规范（OpenHands + AgentScope）

1. 统一任务协议：
   - 输入：`goal, context, permission_ticket, risk_level`
   - 输出：`result, evidence, references, error_code, cost`
2. 统一状态机：
   - `pending -> running -> succeeded|failed -> rolled_back(optional)`
3. 统一错误码：
   - `AUTH_*`, `POLICY_*`, `EXEC_*`, `RUNTIME_*`, `COMPLIANCE_*`

## 8. 里程碑拆解（自动推进）

### M1（第1-2周）

1. 打通身份、任务、审计最小闭环
2. 单岗位单任务成功跑通

### M2（第3-6周）

1. 打通父子Agent编排与执行闭环
2. 上线熔断、回滚、审批
3. 形成首版评估报告

### M3（第7-10周）

1. 上线错误学习隔离池与纠偏任务
2. 上线Skill注册、灰度与退役
3. 完成同部门第二岗位复制

### M4（第11-16周）

1. 上线跨岗位复用与能力市场雏形
2. 输出组织级ROI追踪基线

## 9. 当前自动执行输出节奏

1. 每个里程碑输出一份结果文档（完成项、风险项、下阶段动作）
2. 每次输出包含“可验证证据”：指标截图/日志样本/API结果摘要
3. 默认连续推进，除非出现阻塞（权限、依赖、合规）再触发人工决策
