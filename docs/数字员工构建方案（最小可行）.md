# 数字员工构建方案（最小可行，MVG）

## 1. 目的

在不抑制Agent涌现能力的前提下，以最小治理成本快速验证“数字员工可上岗、可达成目标、可持续成长”。

## 2. 适用范围

仅用于首个试点：

1. 单部门
2. 单岗位
3. 单父Agent
4. 小规模临时子Agent

## 3. 最小定义

数字员工在MVG阶段只需满足：

1. 独立账号身份（可追溯）
2. 岗位目标合同（可考核）
3. 可执行任务闭环（可交付）
4. 可回滚与可停机（可控）

## 4. 最小能力配置

只给原子能力，不预设完整业务技能：

1. Bash执行
2. 文件读写
3. 内外部检索
4. 工具/API调用
5. 测试与日志记录

## 5. 双速机制（防过度设计）

### 5.1 探索通道（快）

1. 允许快速试验
2. 允许临时子Agent
3. 允许引入候选开源能力（仅沙箱）

### 5.2 生产通道（严）

1. 高风险动作必须审批
2. 不可篡改审计
3. 达标后才能转正沉淀

## 6. 最小治理控制（仅4项硬控制）

1. 最小权限：默认拒绝，按岗位放权
2. 审计留痕：所有关键动作可追溯
3. 熔断回滚：异常立即降级或停机
4. 红线禁行：越权、敏感外发、未审查生产变更

## 7. 最小防膨胀策略

1. 子Agent：设置并发上限 + TTL自动回收
2. Skill：仅沉淀高复用能力，其他保留为实验资产
3. 开源：仅沙箱验证通过后才允许生产引入

## 8. 错误学习最小闭环

1. 错误样本入隔离池，不直接学习
2. 仅对P2/P3进行受控纠偏
3. P1只做止损和规则加固
4. 回归通过后灰度发布，失败自动回滚

## 9. 最小评估指标（只看3个）

1. 目标达成率（结果）
2. 错误复发率（质量）
3. 有效Skill复用率（成长）

> 说明：前期不扩展复杂指标体系，避免“报表化”抑制涌现。

## 10. 试点节奏（8-12周）

### 周期1：上岗与基线

1. 完成身份与权限接入
2. 打通单岗位任务闭环
3. 建立最小审计与熔断机制

### 周期2：纠偏与沉淀

1. 运行错误学习闭环
2. 沉淀首批高价值Skill
3. 明确可复用能力和退役能力

### 周期3：扩展决策

1. 若指标达标，复制到同部门第二岗位
2. 若未达标，继续收敛能力和治理参数

## 11. 进入完整方案的门槛

满足以下条件后再切换“完整治理方案”：

1. 连续两个周期目标达成率达标
2. 无P1事故
3. 错误复发率明显下降
4. Skill复用开始稳定增长

## 12. 反模式（必须避免）

1. 试点阶段一次性引入全套委员会和重流程
2. 试点阶段同时追求全岗位覆盖
3. 将探索资产直接作为生产资产长期运行
4. 只看能力增长，不看业务结果与风险

## 13. 产品形态（基于本方案）

最小可行阶段的产品形态不是“大而全平台”，而是“一套可上岗的数字员工操作台 + 受控运行时”：

1. 数字员工工作台（业务侧）
   - 岗位配置：部门、岗位、目标、边界、考核指标
   - 上岗看板：任务队列、完成情况、异常告警
   - 审批入口：高风险动作审批、紧急接管/停机

2. 能力装配台（工程侧）
   - 原子能力管理：bash、文件、检索、工具调用
   - Skill沉淀：实验Skill、正式Skill、退役Skill
   - 子Agent策略：创建阈值、TTL、配额

3. 治理控制台（治理侧）
   - 权限中心：最小权限、临时授权、到期回收
   - 审计中心：意图、调用链、输出证据
   - 风控中心：越权检测、失败激增、自动熔断

4. 运行时内核（系统侧）
   - 父Agent调度器：任务分解、路径选择、子Agent调度
   - 执行沙箱：开源引入验证、流程回放、回归测试
   - 评估引擎：目标达成率、复发率、复用率计算

> 形态重点：先做“单岗位可闭环产品”，再扩展成“多岗位数字员工工厂”。

## 14. 自举循环落地路径与阶段性目标

采用“任务驱动 -> 评估纠偏 -> 能力沉淀 -> 再执行”的自举循环，分四阶段推进。

### 阶段S0：种子上岗（0-2周）

目标：让数字员工具备“可控执行”能力。

1. 完成独立身份、权限接入、岗位合同配置
2. 接入原子能力与最小审计链路
3. 打通单任务端到端执行

验收：

1. 可稳定完成基础任务链路
2. 审计日志完整可追溯
3. 支持人工一键接管/停机

### 阶段S1：受控执行（第3-6周）

目标：在真实工作流中达到可用水位。

1. 上线单岗位核心任务集
2. 建立错误分级与隔离池
3. 启用最小熔断与回滚机制

验收：

1. 目标达成率达到预设基线（如70%+）
2. P1事故为0
3. 错误复发率开始下降

### 阶段S2：纠偏沉淀（第7-10周）

目标：形成“越做越好”的能力增益。

1. 对P2/P3错误执行纠偏与负样本学习
2. 沉淀首批高复用Skill并进行灰度发布
3. 启用子Agent受控创建与自动回收

验收：

1. 有效Skill复用率持续上升
2. 回滚率可控且下降
3. 单位任务成本出现下降趋势

### 阶段S3：模板复制（第11-16周）

目标：从“单点成功”进入“可复制成功”。

1. 固化岗位模板（岗位合同+能力包+治理规则）
2. 复制到同部门第二岗位
3. 建立最小跨岗位能力复用机制

验收：

1. 第二岗位上线周期显著缩短
2. 共性Skill复用率达到目标
3. 无新增高风险治理问题

## 15. 从最小可行到完整方案的切换条件

满足以下条件再进入完整治理方案，避免过早过度设计：

1. 连续两个周期目标达成率达标（建议85%+）
2. 连续两个周期P1事故为0
3. 错误复发率下降且趋稳
4. Skill沉淀与复用率达到正向飞轮迹象
5. 同部门至少两个岗位验证可复制

## 16. 基础设施补充：OpenHands + AgentScope

为保证方案可快速落地且后续可扩展，建议采用“双引擎基座”：

1. OpenHands：单Agent任务执行引擎（代码/脚本/文件/命令类工作）
2. AgentScope：多Agent编排与协作引擎（父子Agent协同、消息路由、会话管理）

### 16.1 职责分工

1. OpenHands职责
   - 提供受控执行环境（命令、文件、任务动作）
   - 承担“执行层”闭环：计划 -> 执行 -> 结果产出
   - 作为Skill运行容器承载具体能力

2. AgentScope职责
   - 提供父Agent-子Agent协作框架
   - 管理Agent间通信、任务分派、状态追踪
   - 对接观测与评估数据，为自举循环提供上下文

3. 平台治理层职责
   - 权限、审计、风控、审批保持平台统一，不绑定具体引擎
   - 通过适配层屏蔽底层引擎差异，避免技术锁定

### 16.2 最小集成架构（MVG）

`业务任务 -> 父Agent(AgentScope) -> 子任务分派 -> 执行Agent(OpenHands) -> 结果回传 -> 评估/审计`

关键实现要求：

1. 为OpenHands执行环境增加权限沙箱与命令白名单
2. 为AgentScope会话增加任务ID和审计ID贯通
3. 对父Agent决策与子Agent执行分别打点，支持问题定位

### 16.3 自举循环映射

1. S0阶段：完成OpenHands单任务执行链路 + AgentScope最小编排链路
2. S1阶段：用AgentScope承载错误分级流转，用OpenHands执行纠偏动作
3. S2阶段：在OpenHands沉淀可复用Skill，在AgentScope优化调度策略
4. S3阶段：复用编排模板与执行模板，复制到同部门第二岗位

### 16.4 技术风险与控制

1. 风险：双引擎复杂度提升
   - 控制：统一任务协议（输入、输出、状态、错误码）
2. 风险：执行层与编排层耦合
   - 控制：建立适配层，禁止业务逻辑直接依赖引擎内部实现
3. 风险：可观测性割裂
   - 控制：统一trace_id / task_id / agent_id日志模型

### 16.5 选型原则

1. 先可用后完美：优先跑通单岗闭环，不先追求全能力覆盖
2. 先约束后扩展：先接入治理底线，再扩大Agent自主度
3. 先模板后规模：先固化“岗位模板+编排模板+执行模板”，再复制扩展
