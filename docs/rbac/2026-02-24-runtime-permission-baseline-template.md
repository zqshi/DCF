# 数字员工运行权限基线模板（默认放行，敏感审批）

## 1. 目标

在“数字员工可真实上岗”的前提下，采用“默认放行 + 敏感审批 + 全链路审计 + 自动降级”的治理策略，保证效率与风险平衡。

适用范围：

1. 本机执行（文件、命令、进程、工具调用）
2. 外网访问（检索、API 调用、第三方系统接入）
3. DCF 任务执行链路（任务创建、审批、执行、回滚）

## 2. 总策略

1. 默认放行：非敏感行为自动执行，不阻塞。
2. 敏感审批：命中敏感规则必须进入审批流后执行。
3. 事后可追溯：关键动作写入审计，包含 `trace_id`、`task_id`、`employee_id`。
4. 异常可止损：触发 P1 风险时自动熔断并支持回滚。

## 3. 风险级别与审批映射

对齐当前实现的 `L1-L4`：

1. `L1`: 自动放行（0 审批）
2. `L2`: 自动放行（0 审批）
3. `L3`: 自动放行（0 审批）
4. `L4`: 敏感审批（默认 2 人审批，且角色去重）

默认审批策略（建议与现状保持一致）：

- `L4.requiredApprovals = 2`
- `L4.requiredAnyRoles = ["auditor", "super_admin"]`
- `L4.distinctRoles = true`

## 4. 敏感操作清单（需审批）

### 4.1 系统与环境类

1. 系统级配置变更（系统服务、关键环境变量、权限模型）
2. 影响运行稳定性的进程/网络关键操作（停核心服务、改网关策略）

### 4.2 数据与隐私类

1. 密钥、凭证、令牌读写与导出
2. 含隐私/敏感数据的对外发送与共享

### 4.3 生产变更类

1. 生产环境写入（数据库、对象存储、核心业务配置）
2. 对外发布、部署、升级、回滚执行

### 4.4 不可逆破坏类

1. 批量删除、覆盖、重置
2. 无补偿动作的外部系统写操作

## 5. 非敏感操作示例（默认放行）

1. 读取非敏感文件与知识资料
2. 常规代码与脚本执行（不触发敏感规则）
3. 常规检索与外部信息查询
4. 测试、日志分析、报告生成、低风险自动化任务

## 6. 本机与外网权限建议

### 6.1 本机权限（建议默认开启）

1. 项目目录内文件读写
2. 构建、测试、静态检查命令
3. 指定工具链执行（如 `node`、`npm`、`git`、`bash`）

### 6.2 外网权限（建议分层）

1. 默认允许：公开检索与低风险 API
2. 白名单直连：业务必需域名/API
3. 受控访问：涉及敏感数据或写操作的外部系统（审批后放行）

## 7. 审计与追踪要求

每个关键事件至少包含：

1. `trace_id`
2. `task_id`
3. `employee_id`
4. `risk_level`
5. `action`
6. `resource`
7. `result`（success/denied/rolled_back）
8. `operator`（system 或人工审批人）
9. `timestamp`

建议至少记录以下事件：

1. `task.created`
2. `task.approval.required`
3. `task.approved`
4. `permission.requested`
5. `permission.granted`
6. `task.failed`
7. `task.rollback`

## 8. 熔断与降级策略

触发条件（任一满足）：

1. `P1` 风险事件
2. 敏感操作连续失败激增
3. 审计异常或权限越权检测命中

动作：

1. 立即停止高风险执行
2. 降级到只读模式
3. 挂起外部写操作
4. 触发人工接管与回滚流程

## 9. 落地配置模板

参考文件：

- `/Users/zqs/Downloads/project/DCF/app/config/runtime-permission-policy.example.json`

接入步骤（最小）：

1. 在员工配置中更新 `approvalPolicy.byRisk`（L1-L3 自动，L4 审批）
2. 在运行网关配置工具与域名白名单（`OPENCLAW_ALLOWED_TOOLS`、`OPENCLAW_ALLOWED_HOSTS`）
3. 在任务入口写入风险级别与审计字段
4. 在管理端演练一次审批、拒绝、回滚链路

## 10. 例外说明

当前模板采用“默认放行”而非“默认拒绝”，这是为匹配“助理型数字员工高效率执行”的业务目标做的阶段性策略例外。  
例外边界：仅对非敏感动作开放；敏感动作仍维持强审批与审计约束。
