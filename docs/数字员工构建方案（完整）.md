# 数字员工构建方案（完整）

## 1. 方案目标

本方案用于在公司内构建“真实上岗、可治理、可涌现成长”的数字员工体系。核心目标：

1. 数字员工以真实组织成员身份上岗（独立账号、真实权限流程、可追责）。
2. 数字员工从原子能力出发，在机制约束下通过任务驱动涌现成长。
3. 形成父Agent-子Agent协作生态，实现岗位能力覆盖和自运营能力。
4. 通过分阶段复制，最终具备覆盖公司全岗位的可行路径。

## 2. 基本定义与边界

### 2.1 数字员工定义

数字员工是绑定部门与岗位、具备独立身份与权限、能够在授权范围内独立完成业务目标、可被审计和考核、可持续成长的Agent。

### 2.2 与相邻概念区分

1. RPA：固定规则执行，不具备自主能力。
2. Copilot：辅助建议，不独立闭环执行。
3. 普通Agent：可执行任务但缺岗位治理与责任链。
4. 数字员工：岗位化、责任化、治理化、成长化。

### 2.3 涌现原则

不预先设计完整技能树；平台只提供原子能力和治理机制。岗位能力由任务过程中的能力缺口驱动涌现。

## 3. 组织身份与上岗制度

### 3.1 身份要求

每个数字员工必须绑定：

1. `employee_id`
2. 独立账号与凭证
3. 部门与岗位
4. 直属负责人（业务Owner）
5. 技术负责人（技术Owner）
6. 治理负责人（治理Owner）

### 3.2 岗位合同

每个数字员工必须具备岗位合同：

1. 职责范围（Do）
2. 严禁边界（Do Not）
3. KPI目标
4. 升级/降级规则
5. 应急停机规则

## 4. 架构与运行模式

### 4.1 父Agent-子Agent模型

1. 父Agent：负责目标分解、策略选择、风险控制、审计归档。
2. 子Agent：按需创建的临时执行单元，任务完成即回收。

### 4.2 原子能力基座

初始能力仅包含：

1. Shell/Bash执行
2. 文件读写
3. 检索与网络访问
4. 内部工具与流程查询
5. 测试执行与结果解析

### 4.3 任务驱动涌现闭环

`任务输入 -> 能力缺口识别 -> 内部复用检索 -> 开源检索 -> 引入/自建决策 -> 执行 -> 评估 -> 沉淀`

## 5. 能力增长治理（防膨胀）

### 5.1 子Agent治理

1. 并发上限
2. 日创建配额
3. TTL自动销毁
4. 相似功能去重

### 5.2 Skill治理

1. 新增Skill必须通过测试、灰度、审计
2. 低复用Skill进入退役队列
3. 复用率和收益持续低于阈值自动淘汰

### 5.3 开源引入治理

1. 许可证审查
2. 供应链安全扫描（SBOM、漏洞）
3. 维护活跃度评估
4. 可替代性评估

### 5.4 内部能力优先策略

能力获取优先级：

1. 内部工具/流程复用
2. 开源引入
3. 自建能力

## 6. 安全与合规

### 6.1 核心控制

1. 默认拒绝 + 最小权限
2. JIT临时授权
3. 高风险动作双审批
4. 不可篡改审计日志
5. 异常检测与自动熔断

### 6.2 红线控制

1. 禁止未授权访问数据
2. 禁止绕过审批执行高风险操作
3. 禁止未经审查引入外部代码到生产
4. 触发红线自动降级并人工接管

## 7. 错误学习与纠偏

### 7.1 错误学习层定义

错误学习层将失败案例转化为“负样本+规则纠偏”，用于降低复发率。

### 7.2 错误分级

1. P1：安全/合规/重大业务错误
2. P2：业务结果错误
3. P3：体验质量错误

### 7.3 纠偏机制

1. 失败样本先进入隔离池
2. P1仅允许降权、规则加固、人工复核
3. P2/P3可做策略优化与负样本学习
4. 更新前必须通过黄金用例回归
5. 灰度失败自动回滚

## 8. 评估体系（人机同口径）

总分：`目标达成度 70% + Skills沉淀度 30%`

### 8.1 目标达成度

1. 任务完成率
2. SLA时效
3. 质量达标率
4. 业务收益（降本/提效）

### 8.2 Skills沉淀度

1. 新增有效Skill数量
2. Skill复用率
3. Skill稳定性
4. 回滚率（越低越好）

### 8.3 成长阶段

1. T0上岗期：达到创建者60%能力
2. T1对齐期：达到创建者85%-100%能力
3. T2超越期：在效率或覆盖上超越创建者

## 9. 落地路线图

### 阶段A（单岗位试点）

1. 1个岗位
2. 1个父Agent
3. 受控子Agent与最小治理闭环

### 阶段B（同部门复制）

1. 复制岗位模板
2. 构建部门级Skill资产库
3. 强化退役与去重机制

### 阶段C（跨部门扩展）

1. 统一权限与审计中心
2. 建立企业级能力市场
3. 建立跨岗位协作协议

### 阶段D（全域自运营）

1. 岗位覆盖持续提升
2. 风险控制稳定
3. ROI持续为正

## 10. 关键风险与补充要求

1. 法律责任归属：明确企业主体责任与RACI。
2. 供应链风险：建立应急替换与版本冻结机制。
3. 数据合规：数据分级、保留销毁、敏感信息策略。
4. 组织变革：明确人类岗位迁移和监督职责。

## 11. 验收标准

1. 单岗任务达成率 > 85%
2. P1事故 = 0
3. 错误复发率下降趋势明确
4. Skill复用率持续提升
5. 单位任务成本持续下降

## 12. 产品形态（完整方案）

完整方案的产品形态是“数字员工操作系统”，由四个工作面和一个运行内核组成。

### 12.1 业务工作面

1. 岗位中心：岗位模板、岗位合同、部门归属、目标/KPI配置
2. 上岗中心：数字员工入职、转岗、停岗、销户
3. 绩效中心：目标达成、质量趋势、价值贡献

### 12.2 能力工程面

1. 原子能力库：bash、文件、检索、工具调用等基础能力
2. Skill工厂：Skill开发、测试、灰度、发布、退役
3. 子Agent管理：创建策略、配额策略、TTL策略、复用策略

### 12.3 治理合规面

1. 权限中心：最小权限、JIT授权、审批策略、到期回收
2. 审计中心：决策链、调用链、结果链、责任链
3. 风控中心：异常检测、熔断、降级、接管
4. 合规模块：许可证治理、数据治理、供应链治理

### 12.4 生态运营面

1. 能力市场：内部能力检索、复用、评级、成本核算
2. 资产运营：复用率、稳定性、边际收益追踪
3. 生命周期运营：实验资产、正式资产、退役资产闭环管理

### 12.5 运行内核

1. 父Agent控制平面：任务分解、策略编排、风险约束
2. 子Agent执行平面：按需执行、快速回收、结果回传
3. 自举循环引擎：评估、纠偏、沉淀、再执行

## 13. 自举循环落地（完整版阶段目标）

在完整方案中，自举循环不是单点机制，而是组织级持续运行机制。

### 阶段F0：基线搭建（0-1个月）

目标：建立统一对象模型与治理底线。

1. 完成岗位合同、身份体系、权限体系标准化
2. 建立最小审计与熔断能力
3. 建立错误分级和隔离池机制

验收：

1. 单岗位可上岗
2. 审计可追溯
3. 风险可停机

### 阶段F1：单岗闭环（第2-3个月）

目标：验证“任务-纠偏-沉淀”循环可持续。

1. 稳定运行单岗位核心任务
2. 完成首批Skill沉淀与灰度发布
3. 建立P1/P2/P3分级处置机制

验收：

1. 目标达成率达标
2. P1事故为0
3. 错误复发率下降

### 阶段F2：同部门复制（第4-6个月）

目标：从“单点成功”转为“模板成功”。

1. 岗位模板复制到同部门多岗位
2. 复用部门级Skill资产库
3. 建立资产去重与退役机制

验收：

1. 第二岗位上线周期显著缩短
2. 复用率持续提升
3. 新增风险可控

### 阶段F3：跨部门扩张（第7-12个月）

目标：形成企业级数字员工生态雏形。

1. 接入跨部门工具和流程
2. 上线企业级能力市场
3. 统一审计与风控策略

验收：

1. 跨部门任务成功率稳定
2. 成本与收益可量化
3. 合规事件保持低位

### 阶段F4：组织级自运营（12个月+）

目标：数字员工生态进入稳定自运营。

1. 多岗位持续自举优化
2. 形成稳定的能力沉淀与复用飞轮
3. 实现“业务目标达成+风险可控+ROI为正”三重稳定

验收：

1. 岗位覆盖率持续上升
2. 关键业务目标可持续达成
3. 治理成本不随规模线性上升

## 14. 与最小可行方案（MVG）的衔接

### 14.1 MVG进入完整方案的触发条件

1. 连续两个周期目标达成率达标（建议85%+）
2. 连续两个周期P1事故为0
3. 错误复发率下降并趋稳
4. Skill复用率呈持续上升趋势
5. 同部门至少两个岗位验证可复制

### 14.2 衔接原则

1. 先复制成功，再扩治理复杂度
2. 先稳定收益，再扩组织覆盖
3. 先固化模板，再扩大涌现范围

### 14.3 常见衔接失败原因

1. 过早引入重治理流程，压制探索速度
2. 复制前未固化模板，导致跨岗位漂移
3. 只追求能力增长，忽视业务结果和风险

## 15. 基础设施方案：OpenHands + AgentScope

为支持数字员工从试点到规模化，建议采用“执行引擎 + 编排引擎”双层基础设施：

1. OpenHands：执行引擎，负责任务动作落地与工具操作
2. AgentScope：编排引擎，负责多Agent协作、消息路由、状态编排

### 15.1 架构定位

1. OpenHands位于执行平面
   - 承担命令执行、文件操作、脚本运行、结果产出
   - 承载Skill运行时和任务动作执行

2. AgentScope位于控制平面
   - 承担父Agent任务分解、子Agent调度、会话与上下文管理
   - 管理多Agent协作策略和协同拓扑

3. 治理服务横切两层
   - 权限中心、审计中心、风控中心、评估中心统一接入
   - 通过适配协议解耦底层引擎实现

### 15.2 与现有模块映射

1. 父Agent控制平面 -> AgentScope
2. 子Agent执行平面 -> OpenHands
3. Skill工厂 -> OpenHands执行容器 + 平台Skill注册中心
4. 自举循环引擎 -> AgentScope编排事件 + 平台评估引擎
5. 治理合规面 -> 独立平台服务（不下沉到引擎内部）

### 15.3 标准接口约束

建议定义统一任务协议，避免后续扩展受限：

1. 任务输入：目标、上下文、权限票据、风险级别
2. 任务输出：结果、证据、引用、成本、错误分类
3. 状态模型：pending/running/succeeded/failed/rolled_back
4. 追踪字段：trace_id、task_id、parent_agent_id、child_agent_id

### 15.4 与自举阶段对齐

1. F0：完成双引擎最小接入，打通统一日志与审计链路
2. F1：基于AgentScope完成父子协同，基于OpenHands完成执行闭环
3. F2：沉淀可复用执行模板与编排模板，实现同部门复制
4. F3：扩展跨部门协作协议与企业级能力市场对接
5. F4：形成“编排策略自优化 + 执行能力持续沉淀”的稳定飞轮

### 15.5 工程化建议

1. 适配层优先
   - 在平台侧建立Engine Adapter，屏蔽OpenHands/AgentScope细节
2. 可观测优先
   - 统一指标、日志、调用链，避免双引擎观测割裂
3. 失败治理优先
   - 将错误分级、回滚、降级放在编排层统一处理
4. 安全边界优先
   - 将权限校验和审批校验放在执行前置阶段

### 15.6 演进策略

1. 初期：单岗位单编排模板单执行模板
2. 中期：模板参数化，支持多岗位复用
3. 后期：策略学习驱动模板自动优化，进入组织级自运营
