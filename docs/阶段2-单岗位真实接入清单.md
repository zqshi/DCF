# 阶段2：单岗位真实系统接入清单（首批）

目标：把“任务演示闭环”升级为“真实系统可写入闭环”，并控制重复写入与失败补偿风险。

## 1. 岗位与系统范围

1. 锁定单岗位：例如财务对账专员
2. 锁定单系统：例如 ERP
3. 锁定单操作：例如 `invoice.create` 或 `payment.update`

## 2. 接入硬要求

1. 每个外部写操作必须提供 `idempotencyKey`
2. 相同 `employee + system + operation + idempotencyKey` 禁止重复创建任务
3. 回滚触发时，如配置补偿动作，必须生成补偿队列项并留痕审计事件
4. 外部系统调用结果必须保留证据字段（请求摘要、响应摘要、时间戳）

## 3. 演练与验收

1. 幂等演练：重复提交同一幂等键，系统应拒绝第二次提交
2. 补偿演练：触发回滚后应出现 `integration.compensation.queued`
3. 审计验收：关键事件包含 `trace_id`、`task_id`、`employee_id`
4. 通过门禁：
   - `cd /Users/zqs/Downloads/project/DCF/app && npm test`
   - `cd /Users/zqs/Downloads/project/DCF/app && npm run drill:p1-rollback`
   - `cd /Users/zqs/Downloads/project/DCF/app && npm run verify:prod-gates`

## 4. 本轮已落地能力（2026-02-16）

1. 任务支持 `externalWrite` 合约字段（系统、操作、资源、幂等键、补偿动作）
2. 用例层实现外部写操作幂等防重
3. 回滚路径实现补偿入队事件 `integration.compensation.queued`
4. 补偿执行器已接入任务循环，支持 `integration.compensation.running|succeeded|retry_scheduled|dead_lettered|deferred`
5. 新增前台契约测试：非法 `externalWrite` 请求返回 `400`
6. 补偿失败支持指数退避重试，超出上限进入死信（`integration.compensation.dead_lettered`）
7. 管理端支持手工重试补偿：`POST /api/admin/tasks/:taskId/retry-compensation`
